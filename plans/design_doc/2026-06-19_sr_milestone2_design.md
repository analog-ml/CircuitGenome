# Design Doc: SR Milestone 2 — Primitive Patterns & Multi-Level Composition

Status: **design only — no source-code changes in this issue.** This document is
the design addendum for [issue #33](https://github.com/analog-ml/CircuitGenome/issues/33).
Read `plans/design_doc/subcircuit_and_functional_block_recognizer.md` (§1–§5) first
for the MVP architecture that this addendum extends.

---

## 1. Motivation

The MVP SR (issue #31) matches a flat single-level pattern library
(`subcircuit_patterns.yaml`) against circuits generated by CircuitGenome's own
synthesizer. Every pattern is a 1:1 replica of a `opamp_modules.yaml` variant —
a "composite" that maps directly to a topology slot.

Milestone 2 must handle `examples/netlist.ckt`, a hand-written op-amp with
generic net/ref naming. Its expected output, `examples/subcircuits.xml`, reveals
that correct recognition requires **three levels of structure** that the flat
single-pass approach cannot produce:

| Level | Example patterns | Size |
|-------|-----------------|------|
| 0 — leaf | diode-connected MOSFET, standard MOSFET, capacitor | 1 device |
| 1 — structural | cascode pair, current mirror pair, differential pair | 2 devices |
| 2 — functional | cascode current mirror, cascoded diff pair, analog inverter | 3–4 devices, children from level 1 |

The MVP's 34 composite patterns are essentially level-2 patterns without
children. Milestone 2 adds level-0 primitives, level-1 structural composites,
and the engine to compose them, populating `RecognizedStructure.children`
(a field already in the data model, unused until now).

---

## 2. Key Insight from the Ground-Truth Output

`examples/subcircuits.xml` shows a critical property: **sharing is intentional**.
Device `/m16` (a diode-connected NMOS) appears as a child of three separate
parent structures simultaneously:

```
MosfetCascodeCurrentMirror[1]
  └── MosfetDiodeStack[1]
        └── MosfetDiodeArray[2]  ← m16   (reference device of cascode mirror)

MosfetSimpleCurrentMirror[1]
  └── MosfetDiodeArray[1]        ← m16   (reference device of simple mirror #1)

MosfetSimpleCurrentMirror[2]
  └── MosfetDiodeArray[1]        ← m16   (reference device of simple mirror #2)
```

`m16` is the common reference transistor that all three mirrors "read." This is
correct analog circuit topology, not ambiguity. Any algorithm that requires a
primitive to belong to exactly one parent will produce wrong results at
level-1 and above.

**Implication for the algorithm**: exclusive ownership applies only at
**level-0** (one primitive per physical device). At level-1 and above, the same
`RecognizedStructure` instance may appear in the `children` list of multiple
parents. The composition graph is a DAG, not a tree.

## 3. Naming Convention

### Drop the `_Array` / `MosfetXxxArray` suffix

`subcircuits.xml` uses "Array" to reflect an IC layout concept (a single
transistor is a degenerate device array). We do not adopt this:

- None of the 34 MVP patterns use "array" — the project convention is
  function-first snake_case (`differential_pair_nmos`, `active_load_nmos`,
  `diode_connected_mosfet_bias`).
- "Normal Array" is especially misleading — a MOSFET with an unconstrained gate
  is just a MOSFET.
- "Array" adds no semantic value at the pattern level; instance multiplicity is
  already tracked by `RecognizedStructure.index`.

### Level-0 pattern names

| Physical structure | Our name | Reference XML name |
|-------------------|----------|-------------------|
| NMOS, gate=drain | `diode_connected_nmos` | `MosfetDiodeArray` (techType=n) |
| PMOS, gate=drain | `diode_connected_pmos` | `MosfetDiodeArray` (techType=p) |
| NMOS, unconstrained | `nmos` | `MosfetNormalArray` (techType=n) |
| PMOS, unconstrained | `pmos` | `MosfetNormalArray` (techType=p) |
| Resistor | `resistor` | — |
| Capacitor | `capacitor` | `CapacitorArray` |

`diode_connected_nmos` mirrors the existing vocabulary (`diode_connected_mosfet_bias`).
`nmos` / `pmos` are direct and unambiguous for a level-0 catch-all.

### Level-1 structural composite names

| Structure | Our name | Reference XML name |
|-----------|----------|-------------------|
| 2-transistor cascode stack, same type | `cascode_pair_nmos` / `_pmos` | `MosfetCascodePair` |
| 2-transistor diode stack (both diode) | `diode_stack_nmos` / `_pmos` | `MosfetDiodeStack` |
| 2-transistor diff pair | `differential_pair_nmos` / `_pmos` | `MosfetDifferentialPair` |
| 2 MOSFETs, shared gate, different drains | `gate_coupled_pair_nmos` / `_pmos` | `MosfetGateConnectedCouple` |
| Diode + mirror output (simple mirror) | `current_mirror_nmos` / `_pmos` | `MosfetSimpleCurrentMirror` |

Note: `differential_pair_nmos` / `differential_pair_pmos` already exist as
level-2 (MVP composite) patterns. At the level-1 refinement, these same patterns
gain a `children` declaration — they don't need to be renamed.

## 4. YAML Schema Extensions

### 4.1 `config/primitives.yaml` — new file

Single-device, level-0 patterns. No `category` field (they are universal across
circuit families). Two new fields: `exclusive` (level-0 ownership rule) and
`priority` (tiebreak when multiple level-0 patterns match the same device).

```yaml
# config/primitives.yaml
primitives:
  # ── NMOS ──────────────────────────────────────────────────────────────────
  - name: diode_connected_nmos
    devices:
      - {ref: m, type: nmos}
    same_net:
      - [m.d, m.g]              # gate shorted to drain
    pins:
      drain:  m.d
      source: m.s
      bulk:   m.b
    tech_type_from: m
    exclusive: true
    priority: 10                # higher priority → matched before lower-priority patterns

  - name: nmos
    devices:
      - {ref: m, type: nmos}
    pins:
      drain:  m.d
      gate:   m.g
      source: m.s
      bulk:   m.b
    tech_type_from: m
    exclusive: true
    priority: 0                 # catch-all, matched only if no higher-priority pattern fits

  # ── PMOS ──────────────────────────────────────────────────────────────────
  - name: diode_connected_pmos
    devices:
      - {ref: m, type: pmos}
    same_net:
      - [m.d, m.g]
    pins:
      drain:  m.d
      source: m.s
      bulk:   m.b
    tech_type_from: m
    exclusive: true
    priority: 10

  - name: pmos
    devices:
      - {ref: m, type: pmos}
    pins:
      drain:  m.d
      gate:   m.g
      source: m.s
      bulk:   m.b
    tech_type_from: m
    exclusive: true
    priority: 0

  # ── Passives ──────────────────────────────────────────────────────────────
  - name: resistor
    devices:
      - {ref: r, type: resistor}
    pins:
      p: r.p
      n: r.n
    exclusive: true
    priority: 10

  - name: capacitor
    devices:
      - {ref: c, type: capacitor}
    pins:
      plus:  c.plus
      minus: c.minus
    exclusive: true
    priority: 10
```

### 4.2 `config/opamp.yaml` — replaces `subcircuit_patterns.yaml`

All opamp patterns (level-1 structural composites + the existing 34 level-2
functional composites), with a new optional `children` field. The algorithm
derives each pattern's level automatically from the `children` dependency graph
— authors do not declare `level` explicitly.

```yaml
# config/opamp.yaml
patterns:
  # ── Level-1: structural composites ────────────────────────────────────────
  - name: cascode_pair_nmos
    devices:
      - {ref: m_upper, type: nmos}
      - {ref: m_lower, type: nmos}
    same_net:
      - [m_upper.s, m_lower.d]    # cascode stack: upper source = lower drain
    pins:
      drain:  m_upper.d
      inner:  m_lower.d
      source: m_lower.s
      gate1:  m_upper.g           # typically the cascode bias gate
      gate2:  m_lower.g           # typically the main signal gate
    tech_type_from: m_upper
    children:
      - pattern: nmos             # upper transistor → level-0 nmos
        devices: [m_upper]
      - pattern: nmos             # lower transistor → level-0 nmos
        devices: [m_lower]

  - name: diode_stack_nmos
    devices:
      - {ref: m_upper, type: nmos}
      - {ref: m_lower, type: nmos}
    same_net:
      - [m_upper.d, m_upper.g]    # upper is diode-connected
      - [m_upper.s, m_lower.d]    # cascode stack
      - [m_lower.d, m_lower.g]    # lower is diode-connected too
    pins:
      drain:  m_upper.d
      inner:  m_lower.d
      source: m_lower.s
    tech_type_from: m_upper
    children:
      - pattern: diode_connected_nmos
        devices: [m_upper]
      - pattern: diode_connected_nmos
        devices: [m_lower]

  - name: current_mirror_nmos
    devices:
      - {ref: m_ref, type: nmos}
      - {ref: m_out, type: nmos}
    same_net:
      - [m_ref.d, m_ref.g]        # reference branch is diode-connected
      - [m_ref.g, m_out.g]        # shared gate
    pins:
      input:  m_ref.d
      output: m_out.d
      source: m_ref.s
    tech_type_from: m_ref
    children:
      - pattern: diode_connected_nmos
        devices: [m_ref]
      - pattern: nmos
        devices: [m_out]

  # (pmos variants: cascode_pair_pmos, diode_stack_pmos, current_mirror_pmos
  #  — same structure, type: pmos throughout)

  # ── Level-2: functional composites (existing 34 MVP patterns, now with
  #    optional children; children added incrementally as level-1 patterns
  #    are defined) ─────────────────────────────────────────────────────────
  - name: differential_pair_nmos
    category: input_pair
    devices:
      - {ref: m1, type: nmos}
      - {ref: m2, type: nmos}
    same_net:
      - [m1.s, m2.s]
      - [m1.b, m2.b]
    pins:
      in1:  m1.g
      in2:  m2.g
      out1: m1.d
      out2: m2.d
      tail: m1.s
      bulk: m1.b
    tech_type_from: m1
    children:                     # newly added in Milestone 2
      - pattern: nmos
        devices: [m1]
      - pattern: nmos
        devices: [m2]
  # ... (remaining 33 MVP patterns unchanged except for optional children)
```

### 4.3 Schema changes to `PatternDef` dataclass

Two new optional fields added to
`circuitgenome/recognizer/models.py::PatternDef`:

```python
@dataclass
class ChildDef:
    pattern: str            # name of the expected child pattern
    devices: list[str]      # device refs in THIS pattern that form the child

@dataclass
class PatternDef:
    name: str
    category: str | None    # None for level-0 primitives (no opamp category)
    devices: list[PatternDevice]
    same_net: list[list[str]]
    pins: dict[str, str]
    tech_type_from: str | None
    hook: str | None
    # ── Milestone 2 additions ──────────────────────────────────────────────
    children: list[ChildDef] = field(default_factory=list)
    exclusive: bool = False   # level-0 ownership flag
    priority: int = 0         # tiebreak for exclusive patterns
```

`ChildDef` replaces the bare comment "populated by a future milestone" in
`RecognizedStructure.children`.

## 5. Multi-Level Composition Algorithm

### 5.1 Pattern loading and level assignment

On startup, load `primitives.yaml` and `opamp.yaml`. Build a directed dependency
graph where an edge `A → B` means pattern A declares B as a child. Run a
topological sort to assign each pattern a level:

```
level(P) = 0                              if P.children is empty
level(P) = max(level(C.pattern) for C in P.children) + 1   otherwise
```

Level-0 patterns are exactly those without children (`exclusive=True` ones from
`primitives.yaml`). All 34 MVP composite patterns that have no `children` yet
remain level-0-equivalent and are matched in a single pass, preserving full
backward compatibility. As `children` are added to composites over time, their
levels increase automatically.

### 5.2 Pass structure

```
Pass 0 (level-0, exclusive):
  For each device D in the netlist:
    Candidates = [P for P in patterns if P.level == 0 and P matches D]
    Sort by P.priority descending
    Assign the highest-priority match exclusively to D
    device_map[D.ref] = RecognizedStructure(name=P.name, devices=[D])

  Result: device_map  (device_ref → RecognizedStructure, injective)

Pass k (level k, sharing allowed, for k = 1, 2, …, max_level):
  For each pattern P with level(P) == k:
    For each assignment A yielded by _find_assignments(P, netlist.devices):
      Verify children(P, A, level_k_minus_1_results):
        For each ChildDef C in P.children:
          child_devices = {A[ref] for ref in C.devices}
          Find a RecognizedStructure S in level_(k-1)_results where
            S.name == C.pattern and set(S.devices) == child_devices
          If not found: skip this assignment (children constraint failed)
      Resolve pins, apply hook
      Create RecognizedStructure(name=P.name, children=[matched S per ChildDef])
      Append to level_k_results

  Note: the same S may appear in children of multiple parents (sharing).
```

### 5.3 Concrete walkthrough — `examples/netlist.ckt`

Netlist (18 MOSFETs + 2 caps). Relevant excerpt:

```spice
m16 net3 net3 gnd! gnd! nmos   ← d=g=net3: diode-connected
m1  net2 net3 gnd! gnd! nmos   ← g=net3, d=net2: plain NMOS
m2  net4 net3 gnd! gnd! nmos   ← g=net3, d=net4: plain NMOS
m6  net5 net5 net8 net8 nmos   ← d=g=net5: diode-connected
m7  net8 net8 gnd! gnd! nmos   ← d=g=net8: diode-connected
m8  net1 net5 net9 net9 nmos   ← plain, g=net5
m9  net9 net8 gnd! gnd! nmos   ← plain, g=net8
```

**Pass 0 results (level-0 exclusive):**

| Device | Same_net matches? | Level-0 assignment |
|--------|------------------|--------------------|
| m16 | d==g (net3) | `diode_connected_nmos` |
| m1 | d≠g | `nmos` |
| m2 | d≠g | `nmos` |
| m6 | d==g (net5) | `diode_connected_nmos` |
| m7 | d==g (net8) | `diode_connected_nmos` |
| m8 | d≠g | `nmos` |
| m9 | d≠g | `nmos` |

**Pass 1 results (level-1 sharing):**

`current_mirror_nmos` requires:
- `m_ref.d == m_ref.g` (diode-connected) → `device_map[m_ref] == diode_connected_nmos`
- `m_ref.g == m_out.g` (shared gate)
- `device_map[m_out] == nmos`

Candidates found:
- `(m16, m1)`: m16.g=m1.g=net3 ✓ → `current_mirror_nmos[0]`, children=[diode(m16), nmos(m1)]
- `(m16, m2)`: m16.g=m2.g=net3 ✓ → `current_mirror_nmos[1]`, children=[diode(m16), nmos(m2)]

Note: `diode_connected_nmos(m16)` is a child of BOTH — sharing is correct.

`diode_stack_nmos` requires both transistors diode-connected with a cascode connection:
- `(m6, m7)`: m6.d=m6.g=net5 ✓, m7.d=m7.g=net8 ✓, m6.s=net8=m7.d ✓
  → `diode_stack_nmos[0]`, children=[diode(m6), diode(m7)]

`cascode_pair_nmos` requires upper source = lower drain:
- `(m8, m9)`: m8.s=net9=m9.d ✓ → `cascode_pair_nmos[0]`, children=[nmos(m8), nmos(m9)]

**Pass 2 results (level-2 functional):**

`cascode_current_mirror_nmos` (if defined — example level-2 pattern) requires:
- `diode_stack_nmos` child over devices {m6, m7}
- `cascode_pair_nmos` child over devices {m8, m9}
- Connectivity: diode_stack gates feed into cascode_pair gates (net5 → m8.g, net8 → m9.g)

→ `cascode_current_mirror_nmos[0]`, children=[diode_stack(m6,m7), cascode_pair(m8,m9)]

**Layer 2 (FBR — unchanged):**

Top-level `RecognizedStructure` instances from all passes are passed to
`assign_slots`. The existing connectivity-scoring algorithm assigns each to a
topology slot. Exclusive ownership at the functional-block level is enforced
here, as today. The `children` field is carried through transparently.

## 6. What Changes vs. Today

| Component | MVP (today) | Milestone 2 |
|-----------|-------------|-------------|
| Pattern files | `subcircuit_patterns.yaml` (flat, 674 lines) | `primitives.yaml` (level-0, ~30 lines) + `opamp.yaml` (levels 1-2, replaces current file) |
| `PatternDef` | `children=[]`, no `exclusive`, no `priority` | add `children: list[ChildDef]`, `exclusive: bool`, `priority: int` |
| `recognize()` | single flat pass | multi-level: exclusive pass-0, then level-ordered composition passes |
| `RecognizedStructure.children` | always `[]` | populated with child instances |
| `SubcircuitRecognitionResult` | flat list of structures | same type, now contains structures with non-empty children |
| FBR / `assign_slots` | unchanged | unchanged |

### What does NOT change

- `_find_assignments` backtracking — reused at every level unchanged
- `_check_same_net` — reused unchanged
- `_resolve_pins` — reused unchanged
- Hook mechanism — carried forward unchanged
- `ParsedNetlist`, `Device` models — unchanged
- `functional_block_recognizer.py` — unchanged
- All 34 MVP patterns — migrated to `opamp.yaml`; `children` added
  incrementally as level-1 patterns are defined (no behavioral change until
  children are declared)

## 7. Migration Path (Backward Compatibility)

Existing patterns with `children=[]` match exactly as today (the children
verification step is a no-op for empty children lists). The existing 73 round-
trip tests continue to pass unchanged.

Level-1 patterns (`cascode_pair_nmos`, `diode_stack_nmos`, `current_mirror_nmos`,
etc.) are additive — they produce new `RecognizedStructure` instances that FBR
may or may not use. They do not alter existing composite matching.

Adding `children` to an existing level-2 pattern is opt-in and only activates
child verification for new candidates — existing `_find_assignments` results
that have no children still pass when `children=[]`.

## 8. Open Questions (Deferred to Implementation Issue)

1. **`opamp.yaml` file name**: Keep as `subcircuit_patterns.yaml` (to avoid
   renaming the existing path reference in `subcircuit_recognizer.py`) or rename
   to `opamp.yaml`? Renaming is cleaner long-term but breaks the default
   `_PATTERNS_PATH` constant.

2. **Level-1 pattern completeness for `netlist.ckt`**: Which level-1 patterns
   are needed beyond `cascode_pair`, `diode_stack`, and `current_mirror`?
   `differential_pair` and `gate_coupled_pair` (from `subcircuits.xml`) are also
   needed — these overlap with existing level-2 patterns and may already be
   covered by the MVP composite match.

3. **`ChildDef` matching strategy at level-1+**: The algorithm above requires
   an exact device-set match (`set(S.devices) == child_devices`). If a child
   has hook-added extra devices, the device set may not match exactly — handle
   by matching on core devices only (pre-hook device set).

4. **Level-0 `category` field**: Set to `None` or a sentinel string like
   `"primitive"` for level-0 patterns? FBR filters by category — `None` ensures
   level-0 primitives are never accidentally assigned to a topology slot without
   an intervening level-2 composite.
